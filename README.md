# 복근 90일 프로젝트 (ABS90)

시작: 2026-07-11 → 목표: 2026-10-09 (약 13주)

---

## 0. 핵심 원리 (왜 이렇게 짜는가)

> **복근 = 근육(이미 있음) + 낮은 체지방(가려져 있을 뿐).**
> 복근 만들기의 90%는 "덮인 지방을 걷어내는 것" = 식단. 운동은 근육 유지 + 대사량.

그래서 이 프로젝트의 **선행지표의 왕 = 칼로리 적자 + 단백질**이다.

---

## 1. 선행지표 vs 결과지표 (이게 핵심)

**결과지표(Lagging)** 는 내가 직접 못 바꾼다. 오늘 "복근 나와라" 한다고 안 나옴.
**선행지표(Leading)** 는 매일 내가 100% 컨트롤 가능하고, 이게 쌓이면 결과가 따라온다.
→ **결과지표는 '확인'만, 에너지는 전부 선행지표 '실행'에 쏟는다.**

| 구분 | 지표 | 주기 | 목표 |
|---|---|---|---|
| 🎮 **선행(매일 컨트롤)** | 칼로리 적자 | 매일 | 유지-500kcal |
| | 단백질 | 매일 | 체중 × 2g |
| | 턱걸이 개수 | **매일(챌린지)** | 1개라도 무조건 |
| | 벤치 볼륨(kg×회×세트) | 주 2~3회 | 주간 우상향 |
| | 런닝 | 주 3회 | 20~30분 |
| | 수면 / 물 | 매일 | 7h / 2L |
| 📉 **결과(확인만)** | 체중 7일 이동평균 | 매일 재고 주간 판단 | 주 -0.5kg |
| | 허리둘레(배꼽) | **주 1회** | 우하향 |
| | 체지방률 / 사진 | 주 1회 | 우하향 |

---

## 2. 트래킹 주기 — "7일 단위가 맞나?" 에 대한 답

**행동(선행지표)은 매일, 몸(결과지표)은 주간으로 판단한다.**

- **체중**: 매일 아침 공복에 재라. 단, **하루 숫자는 무시**하고 **7일 이동평균**만 본다.
  → 하루 체중은 물·염분·글리코겐·대변으로 ±1~2kg 출렁여서 노이즈. 평균이 진짜 추세.
  (앱이 자동으로 7일 평균 그래프를 그려줌)
- **허리둘레·체지방·사진**: **주 1회, 같은 요일·아침·공복**에. (복근은 허리둘레가 진짜 결과지표)
- **왜 주간이 좋은가**: 지방은 하루에 안 빠진다. 1주면 추세가 노이즈를 이긴다.
  너무 자주(매일) 결과를 판단하면 노이즈에 감정 흔들려서 포기하게 됨.

---

## 3. 목표 역산 (앱이 자동 계산)

- 지방 1kg ≈ 7,700kcal
- 하루 500kcal 적자 → 주 약 **0.45kg** 감량 (근손실 최소화되는 안전 페이스)
- 13주 → 약 **5~6kg 지방 감량** → 대부분 이 구간에서 복근 라인이 드러남
- 예시(남 175cm/75kg/28세): 유지 2,350kcal → 목표섭취 **1,850kcal**, 단백질 **150g**
  → 설정 탭에서 내 정보 넣으면 자동 산출

**단백질을 체중×2g로 지키는 이유**: 적자 상태에서 근육을 지켜야 "말라 보이는" 게 아니라
"선명한" 복근이 된다.

---

## 4. 운동 플랜 (턱걸이·벤치·런닝만)

- **턱걸이 (매일 · 챌린지)**: 매일 무조건. 힘든 날은 1개라도. → 스트릭이 끊기지 않는 게 목적.
  세트 예: 월수금 최대 볼륨 / 화목토일은 가볍게 유지. 총 개수만 매일 기록.
- **벤치프레스 (주 2~3회)**: 5×5 또는 4×8. 볼륨(무게×회×세트) 우상향이 근유지·성장 신호.
- **런닝 (주 3회)**: 20~30분. 지방 산화 + 심폐. 벤치 안 하는 날 배치 추천.

---

## 5. 13주 로드맵

| 주차 | 포커스 | 체크 |
|---|---|---|
| 1~2주 | 습관 세팅. 매일 기록 자체를 몸에 붙인다 | 기록 누락 0일 |
| 3~6주 | 적자 안정화. 체중 7일평균 우하향 시작 | 주 -0.4~0.6kg |
| 7~9주 | 정체 가능 구간. 적자 유지 + 단백질 사수 | 허리둘레로 확인 |
| 10~12주 | 라인 드러남. 사진 비교로 동기부여 | 사진 변화 |
| 13주 | 마무리 평가 | 시작 대비 리뷰 |

**정체(플래토) 오면**: 체중이 아니라 허리둘레·사진을 봐라. 2주 완전 정체면 적자 100kcal만 더.

## 6. 앱 사용법

`public/index.html`이 앱 본체다. 폰에서 쓰려면 (아래 배포) → 홈화면에 추가하면 앱처럼 실행.

- **대시보드**: 90일 진행바+D-day · 스트릭 · 오늘의 목표 체크리스트(5/5 달성 시 축하) · 선행지표 · **남은 목표 & 음식 추천** · 체중추세(목표 도달일 예측) · 이번주 운동 · 최근 7일 리포트
  - **남은 목표 & 추천**: 남은 칼로리·부족 단백질·남은 물을 표시하고, **평소 먹는 음식을 트래킹해 "닭가슴살 ×3 / 바나나 ×2"처럼 현실적인 양으로 추천**(단백질 부족 시 고단백 우선, 칼로리 초과 시 저칼로리 고단백만). 추천을 탭하면 바로 담김
- **기록 탭**: 상단 **날짜 선택기(과거 날짜 아무거나 골라 기록/수정 가능)** · 요일별 운동 추천 · 몸무게/수면 · 물 빠른버튼(+250/500ml) · 턱걸이 · 벤치 · 런닝
  - **지난 기록 수정**: 빼먹은 날 채우기 + 잘못 입력한 값 고치기 모두 가능. 날짜 선택 시 기존값이 자동으로 채워져 덮어쓰기·삭제할 수 있고, 캘린더에서 날짜를 탭해 "✏️ 이 날짜 기록/수정"으로도 진입. 과거 편집은 오늘 대시보드에 영향 없음(수정 중엔 상단 배너 표시)
- **캘린더**: 월별 기록 현황(달성한 날 초록 강조), 날짜 탭 → 그날 상세(슬롯별 식단·운동·체중), 누적 기록(턱걸이·런닝·벤치 PR)
- **주간**: FeelFit 2페이지·3페이지 체성분 PDF 자동 판별 및 숫자 인식(저장 전 확인·수정) · 체중/체지방/골격근/내장지방 추이 · 허리둘레 · **바디 프로필 사진**(주간 첨부 → Before/After 자동 비교, 폰에만 저장)
- **빠른 추가**: 최근 먹은 음식 원탭 담기 · 어제 식단 그대로 불러오기
- **설정**: 목표 자동계산 · 다크/라이트 테마 · 내 음식 관리 · 기록 공유 링크(사진 제외) · 백업/복원
- **오늘기록**: 몸무게/물/수면, 턱걸이, 벤치(세트별), 런닝
- **식단**: 아침/점심/저녁/간식 구분 → 음식명 입력 → **DB 기반 자동 칼로리/단백질** (직장인 점심·배달·편의점·마트 380여종)
  - DB에 없으면 **직접 등록**(이름/kcal/단백질) → 저장되어 다음부터 "내음식"으로 검색에 자동 표시 (설정에서 수정·삭제)
- **주간**: 허리둘레·체지방 주 1회 기록
- **설정**: 내 정보 → 목표 자동계산, 백업/복원

일반 기록과 PDF에서 확인 후 저장한 체성분 수치는 브라우저에 즉시 저장한 뒤 로그인 사용자별 Cloudflare D1로 자동 동기화된다. PDF 원본은 서버에 올리지 않으며, 오프라인에서도 기록할 수 있고 연결이 복구되면 서버에 반영된다. 바디 사진은 현재 기기에만 저장된다.

### 개발

- 의존성 설치: `npm install`
- 로컬 D1 적용: `npm run db:local`
- 개발 서버: `npm run dev` → `http://localhost:8792`
- `.dev.vars`의 로컬 사용자로 Access 인증을 대체해 개발한다.

### 배포 (Cloudflare Workers + D1 + Access)

1. `npx wrangler login`
2. `npx wrangler d1 create abs90-db` 실행 후 출력된 `database_id`를 `wrangler.jsonc`에 입력
3. Cloudflare Access 애플리케이션의 팀 도메인과 AUD 값을 `TEAM_DOMAIN`, `ACCESS_AUD`에 입력
4. `npm run db:remote`
5. `npm run deploy`
6. Worker Custom Domain을 `fit.get1004.com`으로 지정하고 Cloudflare Access 이메일 허용 정책 적용

Worker는 `Cf-Access-Jwt-Assertion`을 검증한 뒤 JWT 사용자 ID별로 `user_state`를 분리한다. PC와 폰은 같은 이메일 사용자로 인증하면 같은 기록을 조회하고 저장한다. D1에는 사진을 제외한 프로필·일별 식단/운동·주간 측정·사용자 음식이 JSON으로 저장되며, 버전 번호로 동시 수정 충돌을 감지한다.
